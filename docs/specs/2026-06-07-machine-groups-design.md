# 機器分組（Machine Groups）設計

日期：2026-06-07
狀態：已與使用者確認

## 目標

讓機器可以分組（例：個人、工作、專案），並在軟體 / 拓樸 / 機器三頁以全域切換器過濾視圖，快速聚焦單一情境。

## 需求決策（已確認）

| 決策點 | 結論 |
|---|---|
| 資料模型 | 一台機器屬於單一群組（非多標籤） |
| 群組管理 | 輕量 — 群組由機器欄位帶出，無獨立群組實體/CRUD |
| 切換器 UI | 頂部導覽列右側的全域 segmented switcher |
| 狀態保存 | 全域記住（localStorage），跨頁一致 |
| VM 歸屬 | 預設繼承宿主機，可獨立覆寫 |
| 生效範圍 | 軟體/拓樸/機器三頁過濾；管理頁永遠顯示全部 |
| 過濾位置 | 前端過濾（API 照常回傳全量，多帶群組欄位） |

## 1. 資料模型與 API

### Schema

- `systems` 表新增欄位 `grp TEXT NOT NULL DEFAULT ''`
  - 欄位名用 `grp`（`group` 是 SQL 保留字）；JSON 對外名稱為 `"group"`
  - 沿用 store.go 既有防禦式 migration：`ALTER TABLE systems ADD COLUMN grp TEXT NOT NULL DEFAULT ''`，忽略 "duplicate column name" 錯誤
- `''`（空字串）= 未分組
- 不建群組清單表；群組集合 = 各機器 `grp` 的 distinct 非空值

### 有效群組（effective group）繼承規則

讀取時計算，不落地：

1. `grp != ''` → 用自己的群組（覆寫）
2. `grp == ''` 且 `kind == "vm"` 且 `host_id` 指向存在的 host → 有效群組 = host 的有效群組
3. 其餘（含 host 被刪除、`host_id` 懸空）→ 未分組（`''`）

host 改群組時，未覆寫的 VM 自動跟著變。

### API 變更

1. 回傳 systems 的端點，每台機器多帶兩個欄位：
   - `"group"`：自己存的原始值（管理頁編輯用）
   - `"effective_group"`：含繼承計算後的值（三頁過濾用）
2. `PATCH /api/systems/{id}` 接受 optional `"group"` 欄位（比照現有 `role` 的 `*string` 寫法）；空字串合法（= 清除覆寫/未分組）
3. 不加 `?group=` 查詢參數 — 過濾完全在前端

### 群組名驗證（PATCH 時）

- trim 前後空白
- 長度上限 64 字元（trim 後）
- 其餘不限制（中文 OK）

## 2. 前端：切換器與各頁過濾

### 切換器

- 出現在**軟體 / 拓樸 / 機器**三頁的頂部導覽列右側；管理頁不顯示（不適用）
- 選項動態產生：`全部` + 各機器 `effective_group` 的 distinct 值（依 `localeCompare` 字母序）+ `未分組`（僅當有未分組機器時出現）
- 全部機器都未分組時，切換器整個隱藏（功能零干擾、向後相容）
- 選擇存 `localStorage["cockpit-machine-group"]`
  - 注意：`cockpit-group` 已被軟體頁顯示模式占用，必須避開
  - 存的群組已不存在時自動退回 `全部`
- 切換不重新 fetch，直接以現有資料重渲染

### 各頁過濾規則（以 `effective_group` 比對）

| 頁面 | 行為 |
|---|---|
| 軟體（index） | 只顯示「至少有一台所在群組機器安裝」的軟體列；列內的安裝資訊也只顯示群組內機器；機器篩選下拉只列群組內機器 |
| 拓樸 | 只渲染群組內機器卡片；VM 被覆寫到別群組（host 不在當前群組）時，該 VM 升為頂層獨立卡片 |
| 機器 | 機器切換下拉只列群組內機器；目前選中機器不在群組內時，自動跳到群組內第一台 |
| 管理 | 不過濾；表格加「群組」欄；編輯 modal 加群組輸入 |

### 管理頁的群組編輯

- 編輯 modal 在現有「角色」欄位旁加「群組」欄位
- 形式：可輸入的 datalist 下拉 — 列出既有群組供選，也可直接輸入新名稱
- VM 的群組欄位留空時顯示提示「繼承自 <host 名稱>」

## 3. 邊緣情境與錯誤處理

- **全部未分組**：切換器隱藏
- **選中群組消失**（被改名/清空）：下次載入或資料刷新時退回 `全部`
- **SSE / 輪詢即時更新**：重渲染一律經過同一個過濾函式，更新不會讓被過濾的機器閃回來
- **VM 的 host 被刪除**：繼承失敗 → 視為未分組
- **PATCH 失敗**（400/409）：沿用管理頁現有錯誤提示樣式
- **舊資料寬容**：前端過濾函式對缺 `effective_group` 的資料視為未分組，不噴錯

## 非目標（YAGNI）

- 多標籤（tags）
- 獨立群組實體（顏色、排序、空群組）
- 後端 `?group=` 過濾參數
- 管理頁跟隨群組過濾
